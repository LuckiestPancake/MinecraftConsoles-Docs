29/* API Design considerations
30
31 1. All functions will return S_OK if they succeed or either E_FAIL or an HRESULT error code
32 if they fail. Callers should use the standard SUCCEEDED(hr) or FAILED(hr) macros to
33 test the results, as it eliminates errors due to misconceptions about what exactly is
34 an HRESULT error vs. success/status.
35
36 PLEASE NOTE: S_FALSE is for successfully returning a false, and does *not* qualify as an
37 error code. It should not be used for fail cases. Use only codes like E_FAIL,
38 E_OUTOFMEMORY, E_INVALIDARG, E_POINTER, etc. for failures. Most error codes contain "E_"
39 at or near the start of the name, but you can always tell by whether the top bit is set.
40
41 2. The API is not thread safe at this time, so all calls must be made from the same thread.
42
43 3. Many functions are designed for asynchronous use, and will internally create a task that
44 will only finish after feedback from the server.
45 a. These functions can be either blocking or asynchronous. If the callback pointer is nullptr
46 then it is a blocking call.
47 b. If the call is asynchronous, then it will always return instantly, and will return S_OK
48 if a task has been created.
49 1. If the task is created then at some point the callback will be called once the task
50 finishes, regardless of whether the task succeeds or fails. The task result is then
51 passed into the callback
52 2. However, if the call fails to create a task (due to bad parameters or something else)
53 and returns something other than S_OK, it will *not* call the callback.
54 c. If the call is blocking, the return value will be the task result value.
55 d. The recommended usage is to use async calls, but if a game run all Sentient calls from a
56 worker thread then using blocking calls could work.
57
58 4. The order in which async calls complete is not guaranteed.
59 a. Similarly, if one or more async calls are started, and a blocking call is performed, the
60 async calls may or may not finish before the blocking call returns ( as the blocking call
61 just calls SentientUpdate() until it completes ).
62
63 5. All async functions that take pointers will assume that the pointers point valid memory for the
64 duration of the task, and that the game does not free or overwrite the memory until the task
65 is completed.
66
67 6. All output variables (but not output buffers) are cleared even if the function call fails.
68
69 7. Functions that return buffers of data, will have to have the buffer allocated before being called.
70 NOTE: This is changed from the first release (october 2010)
71
72 8. There are multiple sets of functionality that relies on "blobs" - a blob is a custom (usually
73 game-specific) chunk of binary data that the Sentient does not know anything about.
74*/
